Mybatis 工作流程

无论是 Mybatis 还是 Hibernate，都是 ORM 的一种实现框架（即通过实例对象的语法，完成关系型数据库的操作的技术，也就是所谓的把数据库映射为对象），是对 JDBC 的一种封装。

  • 通过 Reader 对象读取配置文件
  • 通过 SqlSessionFactoryBuilder 对象创建 SqlSessionFactroy 对象
  • 获取当前线程的 SQLSession
  • 事务默认开启
  • 通过 SQLSession 读取映射文件中的操作编号，从而读取 SQL 语句
  • 提交事务
  • 关闭资源

总结：

  • MyBatis 的准备工作与 Hibernate 相似，都需要一个总配置文件、一个映射文件
  • MyBatis 的 SQLSession 工具类使用 ThreadLocal 来对线程中的 Session 进行管理
  • MyBatis 的事务默认开启，需要手动提交
  • 需要手写 SQL 语句，在程序中再通过 映射文件的命名空间 . sql 语句的id 来进行调用
  • 如果需要传入多个参数，那么需要在映射文件中用 map 进行接收
  • MyBatis 支持判断标签，能够通过这些标签完成动态 CURD 的操作

MyBatis 分页

几种分页方式：

  1. 原生 SQL 的Limit分页
  2. Mybatis 自带的 RowBounds 分页
  3. 自定义拦截器插件进行分页
  4. 使用 PageHelper 进行分页

原生 SQL 的 limit 分页

原生Limit分页就是在编写sql语句时需要自己加上limit关键字，然后传入分页参数进行分页，例如select * from t_user limit 0,3

RowBounds 分页

Mybatis内置了一个专门处理分页的类——RowBounds，我们使用它可以轻松完成分页，只需要定义接口方法并完成 sql 映射即可

使用RowBounds分页我们可以不写在映射SQL中写limit关键字，到时候自动回给我们拼接，非常方便，但是该方法处理数据量少时还可以，但是数据量大时，就不行好用了，此时一般都会实现拦截器来完成分页。

自定义拦截器插件分页

需要自己定义一个类实现Interceptor接口，这个接口是Mybatis提供的。任何分页插件想要对Mybatis进行分页就必须实现Interceptor接口，包括后面PageHelper分页插件。

PageHelper 分页插件

PageHelper分页其实也是自定义拦截器方式的一种第三方实现，它内部帮助我们实现了Interceptor的功能。所以实际上我们在执行查询方法之前，PageHelper分页插件同样是对我们的 sql 进行拦截，然后对分页参数进行拼接。

占位符 #{} 和 拼接符 ${} 有什么区别？

#{} 占位符

  1. 占位符可以设置参数，如果传进来的是基本类型，那么其中的变量名可以随意填写，与传进来的参数名可以不一致
  2. 如果传进来的是 pojo 类型（简单 Java 对象），那么其中的变量名必须是 pojo 的属性名。这是因为框架在运行时，需要通过反射根据 #{} 中的名字，将其值放入 sql 语句中，如果占位符中的名称与属性不一致，那么会报错
  3. 不能解决的三类问题：
    1. 动态表名
    2. 动态列名
    3. 动态排序列，如 select * from table order by #{column} 是不正确的

${} 拼接符

  1. 如果传进来的是基本类型，那么其中的变量名必须是 value
  2. 如果传进来的是 pojo 类型，那么其中的变量名必须是 pojo 的属性名
  3. 是字符串连接符，可以用来设置表名、列名、排序名
  4. 可以作为连接符使用，但是不够安全，容易发送 sql 注入问题

两者的区别

  1. 能使用 #{} 就尽量使用，最好不用 ${}

  2. 1#{} 相当于 jdbc 中的 preparedstatement （预编译），${} 是直接使用里面的值进行了拼接。所谓预编译的意思是，例如将一个 #{name} 传进来，预编译是先把要用到的 sql 语句编译为了模板，假设这个 sql 是要查询姓名，那么无论传递的值内容是什么，都之后根据姓名那一列进行查询，无论里面写的是什么都只会被当做一个字符串。

  3. ${} 则与此不同，是在将语句进行拼接之后才确定所要查询的条件/类型的，因此就会有被注入攻击的可能。

  4. 但是对于 order by 是不能用 #{} 的，因为会被替换为字符串自动加引号，这样语句就无法生效

    为了在使用 ${} 时确保安全，需要进行相关的限制，例如不允许用户输入字段或者自行进行转义检查等。

Mybatis 与 Mybatis-plus 有什么区别？

Mybatis 作为一种操作数据库的框架，提供一种 Mapper 类，支持用 Java 代码进行 CURD 的数据库操作，但是完成这一步首先要在 xml 中写好 sql 语句。

Mybatis Generator 能够自动为 Mybatis 生成简单的 sql 语句。

Mybatis 的一个缺点就是 sql 的工作量很大，尤其是在字段较多时，同时其单表操作也非常麻烦，不得不写很多的动态 sql 来兼容查询条件。在使用 MyBatis 时，每当要写一个业务逻辑时都要在 DAO 层写一个方法，再对应一个 sql，即使是简单的条件查询、仅仅改变了一个条件也要在 DAO 层新增一个方法。

Mybatis-plus 是 Mybatis 的一款增强工具，扩展了一些功能，提高了效率。Mybatis-plus 的主要优点如下：

  1. 依赖少：仅仅依赖 Mybatis 以及 Mybatis-Spring
  2. 损耗小：启动就会自动注入基本的 CRUD，性能基本无损耗，直接面向对象操作
  3. 预防 sql 注入：内置了 sql 注入剥离器，能够有效预防 sql 注入攻击
  4. 通用 CRUD 操作：内置通用 Mapper、通用 Service，仅仅通过少量的配置就能够实现单表大部分的 CRUD 操作，还具有强大的条件构造器，满足各类使用需求。

Mybatis 缓存

每次查询数据请求数据库，数据库都会编译并执行 sql 语句，这样效率较低。针对一些频繁访问并不经常改变的数据，并且数据量较大的情况，可以将其加入缓存，每次取之前进行判断，如果缓存不为空，那么就从缓存中取值，如果为空，就请求数据库，并将数据加入缓存

缓存的目的就是，减少和数据库的交互次数，提高执行效率

一级缓存：Mybatis 中 SqlSession 对象的缓存

当执行查询后，查询的结果会同时存入 SqlSession 中的一块区域，当再次查询同样的数据时，Mybatis 会先查询 SqlSession 中是否存在，若存在就不需要再次请求数据库。

SqlSession 表示数据库客户端和数据库服务端之间的一种会话，维护了两者之间的状态信息

第一次发出一个查询 sql，sql 查询结果写入 sqlsession 的一级缓存中，缓存使用 map 作为数据结构，同一个 sqlsession 发出相同的 sql，就从缓存中取，不走数据库。如果两次查询中间出现 commit 操作（修改、添加、删除），本 sqlsession 中的一级缓存区域全部清空，缓存中无法查询数据，需要从数据库查询再写入缓存。

二级缓存：Mybatis 中 SqlSessionFactory 对象的缓存，同一个 SqlSessionFactory 创建的 SqlSession 对象共享缓存

在一级缓存中，最大的共享范围就是一个 sqlsession 内部，如果多个 sqlsession 需要共享缓存，就需要使用二级缓存。开启二级缓存后，会使用 CachingExecutor 装饰 Executor，进入一级缓存的查询流程前，先在 CachingExecutor 进行二级缓存的查询。

二级缓存的范围是mapper级别（mapper同一个命名空间），mapper以命名空间为单位创建缓存数据结构，结构是map<key、value>。每次查询先看是否开启二级缓存，如果开启从二级缓存的数据结构中取缓存数据，如果从二级缓存没有取到，再从一级缓存中找，如果一级缓存也没有，从数据库查询。

sqlsessionfactory 是一个接口，是 sqlsession 工厂，能力就是打开一个 sqlsession 对话，而且重载了许多不同的参数，可以改变这些参数自定义会话过程中的一些默认行为，例如可以设置自动提交事务或关闭自动提交；可以设置获取数据库连接的线程的类型；也可以获取整个 Mybatis 的配置信息 Configuration 对象实例等

SqlSessionFactroyBean 是生产 SqlSessionFactory 的一种工厂 bean

Mybatis 传值方式

单个参数

  • 单个参数的传参比较简单，可以是任意形式的，比如 #{a}、#{b}或者 #{param1}， 但是为了开发规范，尽量使用和入参时一样 。

多个参数

  • 多个参数的情况下有很多种传参的方式，下面一一介绍。

使用索引【不推荐】

  • 多个参数可以使用类似于索引的方式传值，比如 #{param1}对应第一个参数，#{param2}对应第二个参数…….

使用@Param

  • @Param这个注解用于指定key，一旦指定了key，在SQL中即可对应的key入参。

使用Map

  • Mybatis底层就是将入参转换成 Map，入参传Map当然也行，此时 #{key}中的 key就对应Map中的 key。

POJO【推荐】

  • 多个参数可以使用实体类封装，此时对应的 key就是属性名称，注意一定要有 get方法。

List传参

  • List传参也是比较常见的，通常是SQL中的 in。

数组传参

  • 这种方式类似List传参，依旧使用 foreach语法。
