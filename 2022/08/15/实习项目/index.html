断点续传功能怎么实现

由于客户端与服务端的通讯是基于HTTP协议的，所以断点续传功能能够靠HTTP请求来实现。其中最核心的部分就是利用 HTTP 请求中的两个字段：客户端请求头中的 Range 和服务端响应头的 ContentRange

整个过程为：

  1. 客户端请求服务器文件，文件名为 file.zip，服务器域名为 W，请求如下：
    • GET /file.zip HTTP/1.1 浏览器用GET方式获取file.zip文件，HTTP协议版本1.1
    • 指定可接受的响应内容（文件）类型
    • 指定可接受的响应语言
    • 指定可接受的响应内容的编码方式
    • 浏览器身份识别，即浏览器类型验证
    • 指定浏览器想要优先使用的连接类型
  2. 服务器收到请求后，寻找到所请求的文件，提取文件信息，然后返回给浏览器
  3. 此时文件已经开始下载，如果在下载中途中断了下载，那么再次下载文件时从已经中断的地方继续下载。在按下继续下载按钮之后，客户端发送的 HTTP 请求需要包含有一个 range 字段，从而申明本次续传的文件起点。例如设置 Range: bytes = 2000，则是告诉服务器这个文件从 2000 字节处开始传，前面的字节就不用传了。
  4. 服务器收到这个请求之后，返回的信息需要加上 Content-Range 字段，表示已经返回了给定字节的文件数据，同时还会返回文件的全部大小值，例如 Content-Range = bytes 2000-/5999。同时，此时服务端返回的 HTTP 状态码应该是 206，代表已经成功处理了部分 get 请求

在实际场景中还需要考虑一种情况，就是在客户端发起续传请求时，URL 所对应的文件内容可能已经发生了变化，此时再直接进行续传肯定是会出错的。因此需要一个能够标识文件唯一性的方法，可以标识文件的最后修改时间，或者比较文件的唯一标识，如 MD5 值等。

终端在发起请求时也需要在 HTTP 头中申明 if-Match 字段，从而帮助服务端判别文件变化。

run

MDC 是 log4j 提供的一种方便在多线程条件下记录日志的功能。其可以看做是一个与当前线程绑定的哈希表，往其中添加键值对。MDC 中包含的内容可以被同一线程中执行的代码所访问，当前线程的子线程会继承其父线程中的 MDC 中的内容。当需要记录日志时，只需要从 MDC 中获取所需的信息即可。MDC 的内容则由程序在适当的时候保存进去。

run 方法主要负责预处理流程，包括在合适的时间对待测试的任务进行检索，完成 FDFS 文件下载工作，在相应的设备空闲时，从 MQ 中消费对应的任务消息，并将其提交给 ExecuteWorker 来完成实际的测试任务。

    设备存活，将 jobId 传给 preprocessJobId

  9. 将 string 转换为 Json 数据，保存在 prepareTaskInfo 中：

ExecuteWorker.run()

start 方法仅为启动方法，在其中调用了 run 方法

  1. 更新任务状态为执行中：TaskDispatcher.updateTaskInfoStatus，更新设备状态为执行中：deviceMngService.updateExecutingTaskBySerialNumber
  2. 设置标志位 isShutDown 为 false，执行测试：调用方法 taskHandler.doExecuteTester()，若成功则返回成功信息
    1. 若发生 Rollback 异常，则返回执行错误，并 nack 将任务重新放回队列‘
    2. 若发生 DeviceBroken 异常，（应该是判定设备异常），置标志位为 true、返回执行失败信息、关闭任务、nack JobId
    3. 若发生其余异常，则 nack JobId，并 sleep 当前线程 15000 ms
  3. 判断 isShutDown 为 false，则释放设备，并更新设备状态

TaskHandler.doExecuteTester()

该方法中包含几个变量：taskInfo 表示当前任务、preTaskInfo 表示之前任务、testers 表示 注册的tester

  1. 判断是否存在先前任务，弱不存在，则获取合适的 tester（getSuitableTester），并执行 tester.execute，即任务的实际执行方法
  2. 若存在先前任务，则另做处理

TaskTester.execute()

实际执行测试的方法

  1. 检查设备当前状态：checkStatus
  2. 获取任务的 sessionId，并记录当前系统的开始时间
  3. 获取测试任务 getTestTask，对于待测试的 apk 进行一系列处理
  4. 开始任务测试，并置标志位 isContents 为 false，将 taskId 中所存储的每一个 id 与待处理的任务 task 的 id 进行对比，判断该任务是否包含在 taskId 中。若包含则置标志位为 true，开始 app 的测试
  5. times 存储任务的 rollback 次数，当该次数大于等于 4 时，则认为出现兼容性问题，测试任务失败，需要进行进一步的分析
  6. 若未达到最大 rollback 次数，则开始 app 的测试，执行 doExecuteApp 方法
  7. 若出现异常，则重启设备。记录测试相关信息（如时间），并打印日志

TaskTester.doExecuteApp

APP 测试方法

  1. 启动各个子测试任务的线程

  2. 执行动作 perform action，即调用 server.apk 的 start 方法，通过 server.apk 来完成自动化测试工作

  3. 测试结束之后，进行结果数据的获取与处理

    1. stop 同步任务，并记录任务的结束时间、移除对应的任务 id
    2. stop 异步任务，对于每个执行任务的线程，记录任务执行时间，并 stop 对应 task，处理对应的测试结果。

  4. 对于不同任务，stop 方法的实现存在差异，包括日志拉取、截图下载、截图分析、测试异常分析等过程

changeDeviceQueue

修改对应的手机状态

  1. 获取指定设备上的监听队列，将其保存到 deviceKeys 中，将 deviceKeys 中的元素拼接为字符串，并通过 “，” 进行分割。如果拼接过程中，某个 deviceKey 以 “NA” 结尾，且不为 “mtf.empty”，则置标志位 isShared 为 true
  2. 若标志位为 false，则调用 updateExecutingTaskBySerialNumber 方法，其中输入参数为 EXECUTOR_ABANDEN
  3. 若标志位为 true，且信号量中的当前可用许可数 > 0，则调用相同的方法，输入参数为 EXECUTOR_IDLE，同时日志中打印许可数
