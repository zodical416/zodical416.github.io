深拷贝与浅拷贝的区别

浅拷贝

创建一个新对象，该对象有着原始对象属性值的一份精确拷贝。如果属性为基本类型，拷贝的就是基本类型的值，如果属性为引用类型，拷贝的就是内存地址。

深拷贝

将一个对象从内存中完整拷贝一份出来，在堆内存中开辟一个新的区域来存放新对象，并且对新对象进行修改不会影响原对象

301 适合永久重定向，代表所请求的资源已被永久定位到新位置，比较常用的场景是域名跳转。例如对 http://www.baidu.com 的访问会跳转到 https://www.baidu.com

302 是临时重定向，即请求的资源现在临时从不同的 URL 响应请求。适合用来做临时跳转，比如未登录的用户访问用户中心会自动跳转到登录、访问 404 页面会重新定向到首页

总之，302 重定向只是暂时重定向，浏览器会抓取新内容并保留旧内容

301重定向是永久重定向，浏览器抓取新内容的同时将旧网址替换为了重定向后的网址

负载均衡原理

为了避免单机性能瓶颈、解决单点故障的隐患，部署多台 tomcat 服务器，让客户端随机访问其中一台机器。然而如果让客户端来选择所要访问的机器的话，如果有服务器宕机了，客户端是无法提前感知到的，最终导致其连接到已经宕机的服务器上。因此，需要在服务器端前面加一个负载均衡层，一般采用 Nginx 来完成

同时，考虑到这样的架构所有流量最终都会打到 服务器 上，需要进行鉴权，因此又在 Nginx 层后面加了一个 网关 层。这样让网关鉴权通过之后才会将流量转发到服务器，除此之外，网关还能起到风控、协议转换、流量控制的作用。

由于动态请求与静态请求都打到 tomcat，在流量大时会导致其承受不必要的压力，而对于静态资源，tomcat 每次从磁盘加载文件比较影响性能，因此对 Nginx 进行进一步优化，即如果是动态请求，则经过 网关 打到 tomcat，而对于静态请求，则打到静态资源服务器上。这就是所说的动静分离，将动态资源与静态资源分开，这样使 tomcat 专注于处理动态请求，静态请求利用 Nginx 的 proxy cache 等功能，提升处理能力。

HTML 是静态的，ASP、PHP、JSP 等页面是动态的

为了避免 Nginx 发生故障，也需要部署至少两台机器，以主备的形式存在。备 Nginx 会通过 keepalived 机制，即发送心跳包，来及时感知主 Nginx 的存活，发现宕机则自己转为主 Nginx

由于 Nginx 工作在应用层，因此其需要与 客户端 和 服务端 各建立一个 TCP 连接，而 TCP Socket 与 接收、发送缓存区 等都非常消耗内存，这导致连接数量较多时 Nginx 的抗负载能力急剧下降。

为了解决这一问题，又提出了工作在传输层的负载均衡器 LVS。其类似于路由器，只负责转发包而不需要建立连接，因此不需要维护额外的 TCP 连接。

那么 LVS 负载均衡器是如何工作的呢？

负载均衡器在接收到第一个来自客户端的 SYN 请求时，通过负载均衡算法选择一个最佳的服务器，并对报文中的目标 IP 地址进行修改（改为后端服务器 IP），直接转发给该服务器。TCP 的连接建立，也就是三次握手，是由客户端与服务器直接建立的，负载均衡器只是起一个转发作用。为此，可以在 Nginx 上再加一层 LVS，让 LVS 来承接所有流量。

同时，由于 LVS 属于 Linux 的内核模块，工作在内核态，而 Nginx 工作在用户态，因此在性能与稳定性上 Nginx 都不如 LVS。

负载均衡的方法

HTTP 重定向负载均衡

  1. 请求到负载均衡服务器
  2. 根据均衡算法，返回 304，重定向到实际负责计算的服务器 ip，并返回给浏览器
  3. 然后浏览器直接请求实际负责计算的服务器
  4. 最后实际计算的服务器响应到客户端浏览器

缺点：

  1. 请求经过两次服务器(负载均衡服务器、实际计算服务器)，增加了响应时间
  2. 暴露了实际负载的服务器的公网 ip，降低了安全性

DNS 负载均衡

  1. 用户请求域名，请求到 DNS 服务器
  2. DNS 服务返回解析的 IP 地址到客户端，并不暴露给浏览器来进行重定向
  3. 客户端拿到服务器 IP， 对服务器进行请求
  4. 计算服务器返回相关的响应信息

改进：

  1. 不用每次都请求负载的 ip，可以缓存起来，重复使用来提高性能
  2. dns 不必暴露实际计算的服务器 ip，提高安全性

反向代理负载均衡

  1. 客户端发起请求到负载均衡服务器，服务器根据算法得到负载 ip
  2. 负载均衡服务器构造请求，请求内网负载的计算服务器
  3. 计算服务器返回响应结果给负载均衡服务器
  4. 负载均衡服务器返回响应结果给客户端

缺点：基于 http 层做的负载均衡，效率较低；适用于较小的集群

IP 层负载均衡

对网络层 ip 进行替换，不需要在 http 层工作，直接在操作系统内核的 ip 数据包中替换地址，效率比基于 HTTP 层的反向代理更高

  1. 客户端请求负载均衡服务器
  2. 负载均衡服务器修改目的 ip 为内网机器的 ip
  3. 内网机器计算完毕，响应 ip 改为负载均衡服务器 ip 的内网地址
  4. 负载均衡服务器修改响应的 ip 为自己的外网 ip， 返回结果给客户端

缺点：请求和响应都需要经过负载均衡服务器进行 ip 层替换，响应数据会成为后期瓶颈

数据链路层负载均衡

解决响应数据体量过大、效率低的问题，通过修改数据链路层的 Mac 地址，ip 使用虚拟 ip 来实现负载均衡

  1. 客户端请求负载均衡服务器
  2. 负载均衡服务器替换 Mac 地址为计算服务器，ip 为负载均衡服务器ip
  3. 计算服务器直接响应数据给客户端

吞吐量最高

负载均衡算法

常用的负载均衡算法有 轮询、随机、最少链接、源地址散列、加权 等方式

轮询法

将所有请求依次分发到每台服务器上，均衡地对待每一台后端服务器，不关心服务器的连接数与负载情况。轮询的策略目的在于请求的绝对均衡，但是在实际情况下，可能服务器并不完全一样，导致有些性能高的服务器不能完全发挥出来

随机法

根据服务器列表的大小随机获取其中的一台来进行访问，对于完全随机的情况，当调用量增大时，实际的访问效果会越来越接近于轮询

源地址哈希法

思想是获取客户端访问的 ip 地址，通过 hash 函数计算出一个 hash 值，用该 hash 值对服务器列表的大小进行取模，得到的值就是要访问的服务器序号

该方法的好处是，在服务器列表不变的条件下，每次客户端访问的都会是同一个服务器，利用这个特性无需额外操作就可以利用 session 来保存状态

加权轮询法

是对轮询法的优化，考虑到不同的服务器性能不同，所以不能一概而论，需要给性能较低的服务器低权重，性能高的给高权重。

加权随机法

与加权轮询法相似，该方法同样根据后端服务器不同的配置和负载情况来配置不同的权重，不同的是，它根据权重来随机选择服务器而不是按照大小顺序。

最小连接数法

请求次数的均衡并不代表负载的均衡，因此提出了最小连接法，该方法更加灵活和智能。由于后台服务器的配置不尽相同，对于请求的处理有快有慢，因此最小连接法正是根据后端服务器的当前连接情况，动态选取其中当前积压的连接请求数最少的一台来处理当前请求，尽可能提高后台服务器的利用率，将负载合理地分流到每一台服务器。
