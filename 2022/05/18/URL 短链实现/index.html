URL 短链实现

原理

当在浏览器输入短链后：

  1. DNS 首先解析短链前缀地址的 IP 地址
  2. 当 DNS 获得 IP 地址后，会向该地址发送 HTTP GET 请求，查询短码
  3. 服务器会通过短码获得长码的 URL
  4. 通过 302 重定向到长连接

为什么用 302？

这是因为 301 状态码代表永久重定向，浏览器拿到长链后会对其进行缓存，从而无法对短链进行相关分析，例如点击率等

如何生成唯一的短链

也就是如何通过唯一的字符串来表示长链。可以采用哈希算法对长链进行去哈希。一般采用非加密型哈希算法，如 MurmurHash，因为相比于 MD5 等加密型哈希，非加密型的效率更高。

对于生成的哈希值，通常是十进制的，可以将其转为 62 进制，将转换后的值作为短链的唯一标识即可。

如何存储？

可以使用 Redis 这类 K-V 内存数据库来存储，性能更好。

当存放一个长链时，首先判断是否已经被转换过短链。

如果需要对长链进行区分，例如不同用户的长链不同，那么在判断的时候加上对应的条件即可，例如这个长链所对应的用户 ID

如何判断是否发生了哈希冲突？

如果用 MySQL 数据库存储了短链，那么可以给短链字段加上唯一索引。当获得一个短链后，直接将长短链对应关系插入 db 中，如果 db 里不含有此短链的记录，则插入，如果包含了，说明违反了唯一性索引。

为了提高性能并且应对高并发，可以采用布隆过滤器，用所有生成的短网址构建布隆过滤器，当一个新的长链生成短链后，先将此短链在布隆过滤器中进行查找，通过布隆过滤器中的哈希函数将短链映射到位函数上，从而判断是否存在该短链。如果不存在，说明 db 里不存在此短网址，可以插入！

如何解决哈希冲突？

如果发生哈希冲突，就在长链后面再加一个随机的字符串

自增序列算法

除了哈希算法外，还可以通过自增序列算法生成短链。

可以维护一个 ID 自增生成器，比如 1，2，3 这样的整数递增 ID，当收到一个长链转短链的请求时，ID 生成器为其分配一个 ID，再将其转化为 62 进制，拼接到短链域名后面就得到了最终的短网址

主要有以下四种获取 id 的方法

  • 类 UUID

    随机生成全局唯一标识符 UUID，保证在同一时空中所有机器都是唯一的。但这种方式生成的 id 比较长，且无序，在插入 db 时可能会频繁导致页分裂，影响插入性能。

  • Redis

    用 Redis 是个不错的选择，性能好，单机可支撑 10 w+ 请求，足以应付大部分的业务场景，但有人说如果一台机器扛不住呢，可以设置多台嘛，比如我布置 10 台机器，每台机器分别只生成尾号 0，1，2，… 9 的 ID, 每次加 10 即可，只要设置一个 ID 生成器代理随机分配给发号器生成 ID 就行了。

    不过用 Redis 这种方案，需要考虑持久化（短链 ID 总不能一样吧），灾备，成本有点高。

  • Snowflake

  • MySQL 自增主键

    高并发下写的压力很大，为了解决这一问题，可以提前生成自增 id。方案如下：

    设计一个专门的发号表，每插入一条记录，为短链 id 预留 （主键 id * 1000 - 999） 到 (主键 id * 1000) 的号段，例如：

    发号表：url_sender_num

    当长链转短链的请求打到某台机器时，先看这台机器是否分配了短链号段，未分配就往发号表插入一条记录，则这台机器将为短链分配范围在 tmp_start_num 到 tmp_end_num 之间的 id。从tmp_start_num 开始分配，一直分配到 tmp_end_num，如果发号 id 达到了 tmp_end_num，说明这个区间段的 id 已经分配完了，则再往发号表插入一条记录就又获取了一个发号 id 区间。

    解决了发号器问题，接下来就简单了，从发号器拿过来的 id ，即为短链 id，接下来我们再创建一个长短链的映射表即可， 短链 id 即为主键

    不过这里有个需要注意的地方，我们可能需要防止多次相同的长链生成不同的短链 id 这种情况，这就需要每次先根据长链来查找 db 看是否存在相关记录

    一般的做法是给长链加索引，但这样的话索引的空间会很大，所以我们可以对长链适当的压缩，比如 md5，再对长链的 md5 字段做索引，索引就会小很多。这样只要根据长链的 md5 去表里查是否存在相同的记录即可。
