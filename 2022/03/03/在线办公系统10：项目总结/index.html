办公系统 10. 项目总结

发表于 2022-03-03 更新于 2022-03-09 分类于 办公系统
本文字数： 2.4k 阅读时长 ≈ 2 分钟
你说在这样一个夜晚，就算挖空心思也不会看得太远

这是个什么项目？

这个项目是一个用于在线办公的微信小程序，主要功能包括基于微信授权的用户注册与登录、权限管理、进行 GPS 坐标定位并判断所处地区的疫情风险等级、人脸考勤签到以及云文件共享等基本功能。

主要工作是什么？

我主要在这个项目中完成了用户的注册与登录、权限管理与认证、系统消息通知模块的搭建以及坐标定位功能的实现这样几个模块。

数据库的设计

用户的注册与登录功能

首先是用户的注册与登录模块，由于这个项目的运行平台是微信小程序，因此通过微信授权来完成用户的注册以及登录。当用户第一次打开这个微信小程序的时候，会提示需要注册。当点击确认注册后，会请求获得用户微信头像、用户名等相关信息的授权。在得到这一授权之后，微信会给项目后端发送一个临时授权字符串，这个字符串的有效期很短，只能用于这一次的登录或注册，这是出于安全角度考虑的。

在获得这一临时授权字符串之后，后端会向微信发送一个 HTTP 请求，请求包括小程序的 APP id、密钥以及这个临时授权字符串等等信息。微信回复的相应将会包括一个 open ID 字符串，这个字符串是用户的唯一标识。采用这个标识而不是我们所常见的微信 ID 的目的也是为了保护用户的隐私。

在获取到这个 open ID 字符串之后，后端会向用户数据表中查找，是否含有这个字符串，如果不含有，则写入用户信息完成注册，如果包含用户记录，则完成登录功能。

用户的权限管理与认证

为了实现不同用户对应不同的管理权限，采用了经典的 RBAC 权限模型。该模型包括用户、角色与权限三个表，其中权限包括模块表和行为表两个。例如，模块表中包括员工、部门等模块，行为表中包括添加、删除等行为，那么这两个表共同在权限表中组成了添加用户、删除用户等权限。

后端的权限验证交给 Shiro 框架来完成管理，而 Shiro 完成登录认证与授权是借助于 JWT 所生成的 token 字符串的。

为了实现用户连续使用系统、token 不过期的问题，采用了 token 缓存的方案。也就是利用 Redis 存储一个用于续期的 token，该 token 的过期时间比用户客户端保存的 token 要长。

假如当第 15 天时，用户 token 失效，让 Shiro 程序查看 Redis 中是否存在缓存的 token，如果这个 token 不存在，那么证明用户需要重新登录；如果 Redis 中的 token 存在，则直接生成一个新的 token 返回给客户端，并将新 token 保存在 Redis 中。

系统消息通知模块

为了实现系统消息通知模块，选用了 RabbitMQ 作为消息中间件，并利用 MongoDB 存储消息。这是考虑到系统需要记录哪些消息是用户已读、哪些的未读的，这样当用户数量很大时，消息模块需要存储的数据量太大，MySQL 数据库难以支撑。而 MongoDB 就适合存储系统消息这样的海量低价值数据。

同时，由于发送一条系统公告需要向数据库中写入大量数据，MongoDB 也难以支撑，因此引入了消息队列来完成消息通知的削峰填谷。这样整个系统的流程就是：当用户登录系统时，用异步线程从消息队列中接收属于该用户的系统消息，然后再将消息写入数据库中，消息队列中的这条消息就可以删除了。由于用户往往不会同时登录系统，因此这样就把往数据库中写入消息的过程转变为了错峰写入，提高了系统的可用性。

选用 RabbitMQ 的主要原因是它一方面是比较成熟的 MQ，另一方面就是它非常稳定健壮，虽然不如 Kafka 速度快，但是在消息接收这样的应用场景里面还是足够用的。

由于微信小程序不是长连接，因此所创建的消费者对象无需长期存在，在完成消息接收之后就可以销毁。这就需要前端程序设置一个定时器，每过一段时间进行一次轮询请求，来判断是否有新的系统消息需要接收。

坐标定位

在完成系统消息通知模块的基础上，还实现了用户签到时的坐标定位功能。该功能的实现比较简单，可以直接调用小程序获取地理位置的接口方法，获取到的是位置的经纬度，之后利用腾讯位置服务将其转换为具体的地址信息。

获得地址信息之后，利用 本地宝 网站来查询所在地区的疫情风险等级。然而，本地宝并没有提供可调用的 Web 接口，因此只能采用 URL 地址传参的方式获取相应，并从 HTML 响应中解析得到风险等级信息。

URL 需要传入两个参数：城市编码和区县，因此创建了一个数据表，用于存储城市名称与编号。发送请求之后，利用 jsoup 解析 HTML 标签，获取风险等级信息。

还有什么可优化的？

消息通知模块的 MongoDB 存储

随着用户数量的增多、系统消息的累积，MongoDB 所储存的消息将会越来越多，而且通知消息的价值不大，因此可以对 MongoDB 存储系统消息的机制进行优化。

之前我查资料发现，主要有两种机制来应对这种情况，一种就是定期销毁所谓的冷数据，释放内存空间。例如将五年以上的消息数据设为冷数据，并进行定期删除；还有一种就是进行冷热数据的分离，设置两个或多个 MongoDB，通过编写定时程序，在每天凌晨这样用户量少的时刻将冷数据从一个 MongoDB 转移到另一个，这样主数据库的数据量减小，增删改查的速度就变快了，而另一个数据库存放的都是冷数据，即使数据量很大，但是用到的次数不多，仅仅充当归档库。

遇到了哪些问题？

Token 失效问题

当时在采用 token 来进行用户登录验证时，希望能够实现这样一个功能，就是尽管 token 失效的时间为 15 天，但是如果用户在连续登录的情况下，客户端的 token 将一直不会失效，也就是只有用户连续 15 天不登录的时候才会需要重新登录、生成 token。

当时考虑了两种方案，双 token 方案和 Redis 缓存 token 的方案。这两个方案的本质实际上是一样的，双 token 就是客户端持有两个 token，一个是更新用的 token，一个是登录用的，而缓存 token 的方案则是将 token 保存在 Redis 中一份，并设置对应的过期时间。
